Getting Started

Sandbox is a native desktop application. Double-click the Sandbox shortcut or executable to launch it. The app opens in its own window — no browser or terminal needed.

Your settings, tags, favorites, and thumbnails are stored locally and persist between sessions.

Home Page

The home page displays all available miniapps as cards or list items. Use the sidebar to filter by tag, the search bar to find apps by name, and the sort dropdown to change the display order.

Miniapps

Each miniapp is a self-contained tool accessible from the home page. Click any app card to open it. Use the Home button or Close button in the miniapp's top bar to return to the home page.

Tags

Tags categorize your miniapps. Click a tag in the sidebar to filter the home page to only show apps with that tag. Tags are displayed in alphabetical order.

Creating Tags

Click + Add Tag in the sidebar to create a new tag. Enter a name and press Enter or click Add.

Deleting Tags

Hover over a tag in the sidebar and click the X button that appears. You will be asked to confirm. Deleting a tag removes it from all apps that had it assigned.

Assigning Tags to Apps

Open the edit modal for an app (pencil icon) and use the Tags dropdown to check/uncheck tags. Tags in the dropdown are listed alphabetically.

Favorites

Click the star icon on any app card to mark it as a favorite. Favorited apps appear in a dedicated section at the top of the home page when viewing All Apps in Custom Order.

Search and Sort

Use the search bar in the top bar to filter apps by name or description. The sort dropdown lets you order apps by custom order, alphabetical, recently used, or date added.

Card and List View

Toggle between card view (grid of cards with thumbnails) and list view (compact rows) using the view toggle buttons in the top bar. Your preference is saved locally.

Editing App Details

Hover over an app card and click the edit (pencil) icon to open the edit modal. From here you can change the app's title, description, assigned tags, and thumbnail image.

Thumbnails

Each app can have a custom thumbnail image displayed on its card. In the edit modal, click Choose Image... to open a file picker and select an image file (PNG, JPG, GIF, or WebP).

Help Manual

This manual is available at any time from the sidebar. Click Help in the lower sidebar to open it.

FW Allocation

Allocates Field Weld (FW) counts from a pivot table onto Butt Weld (BW) and Socket Weld (SW) labor rows. For each drawing, the tool flips a number of BW/SW rows from Shop (ShopField = 1) to Field (ShopField = 2) equal to the field weld count for that drawing. The largest pipe size gets the first weld (the "header weld"); remaining welds are distributed by longest pipe run footage.

What you need

Two Excel (.xlsx) files, described below. Both must have a header row.

File 1 — FW Count File

A simple pivot table or list showing how many field welds belong to each drawing. The tool reads the first sheet of the workbook.

  • Row 1: header row (any names — contents are ignored)
  • Column A: Drawing Number (text)
  • Column B: Field weld count (integer)
  • Any additional columns are ignored
  • Rows with a blank Drawing Number or a count of 0 are skipped

Example:

Drawing Number   FW Count
ABC-001          3
ABC-002          7
XYZ-101          1

File 2 — Labor File

Your labor estimate file with PIPE, BW, and SW rows. The tool reads the first sheet. Columns can appear in any order, but the following header names must exist exactly (case-sensitive):

  • Drawing Number — text matching the drawing numbers in File 1
  • Component — values used by the tool: PIPE, BW, SW (other components are ignored)
  • Size — numeric pipe size (e.g., 2, 4, 6, 8, 10)
  • Quantity — for PIPE rows, this is the linear footage used to rank welds by pipe run length
  • ShopField — 1 = Shop, 2 = Field. Only rows with ShopField = 1 are eligible to be flipped.

Any other columns (e.g., descriptions, unit labor, area, system) are preserved unchanged in the output.

Step-by-step

  1. Open the FW Allocation miniapp from the home page
  2. Click Choose File... next to "FW Count File" and select your pivot (File 1)
  3. Click Choose File... next to "Labor File" and select your labor estimate (File 2)
  4. Click Run Allocation and wait for processing to finish
  5. Review the Results panel:
    • Welds flipped — total BW/SW rows moved from Shop to Field
    • Drawings processed — drawings in the FW file that matched labor data
    • Issues (shown in orange if any) — drawings the tool could not fully allocate
  6. Click Download Updated Labor and choose a save location. Output is a copy of your labor file with ShopField changed from 1 to 2 on the flipped rows.
  7. Click Download Issues Log (optional) to save a report of drawings that could not be fully allocated.

Allocation logic

For each drawing with a field weld count greater than 0:

  1. Find all BW/SW rows on that drawing with ShopField = 1 (eligible candidates)
  2. Identify the largest pipe size among those candidates. That row gets the first weld (the "header weld"). BW is preferred over SW at the same size.
  3. For remaining welds, order the other sizes by total PIPE footage on the drawing (longest run first), and within each size prefer BW over SW.
  4. Stop when the field weld count is satisfied, or when candidates run out.

Issues log

The Issues Log (FW_Unallocated_Issues.xlsx) contains one row per drawing that could not be fully allocated, with these columns: Drawing Number, FW Count, Flipped, Unallocated, Issue. The possible issues are:

  • NO BW/SW labor rows found — the drawing has a field weld count but no eligible BW or SW rows in the labor file. Check for drawing-number mismatches between the two files, and that the BW/SW rows have ShopField = 1.
  • EXHAUSTED — not enough BW/SW rows — there were eligible candidates but fewer than the field weld count. Verify the FW count or check for missing weld rows in the labor file.

DataLore

Exports every user table from a SQLite database file to Excel and/or CSV. Use it to inspect, share, or archive the contents of a .db file in spreadsheet form.

What you need

A SQLite database file with the extension .db, .sqlite, or .sqlite3. SQLite is a single-file database, so the entire database lives in one file you can pick from disk.

System tables (the sqlite_* bookkeeping tables) are skipped automatically.

Step-by-step

  1. Open the DataLore miniapp from the home page
  2. Click Choose File... and pick a SQLite database file
  3. Choose an output format:
    • Excel (default) — one workbook with a sheet per table
    • CSV — one CSV file per table inside a csv/ subfolder
    • Both — produces both formats
  4. Click Run Export. DataLore writes the output to an internal run folder under your app data directory.
  5. Review the per-table results panel (table name, row count, truncated badge if any).
  6. Click Save As... to copy the output to a location of your choice:
    • Excel-only mode opens a save-file dialog — pick a destination .xlsx path.
    • CSV or Both mode opens a folder picker — the entire run folder is copied into the chosen folder.
  7. Click Reveal in Run Folder to open the original output folder in Windows Explorer.

Type handling

  • Integers and real numbers are written as numeric cells in Excel and as plain numbers in CSV.
  • Text is written as a string cell.
  • NULL values become empty cells.
  • BLOB columns are not extracted — the cell shows <BLOB N bytes> as a placeholder.

Limits

  • Excel sheets cap out at 1,048,576 rows (Microsoft's spreadsheet limit). Tables with more rows are truncated in the Excel output and flagged with a truncated badge in the results. CSV output (if selected) is always complete.
  • Sheet names are sanitised for Excel: characters / \ ? * [ ] : are stripped, and names longer than 31 characters are truncated. If two tables sanitise to the same name, the later sheet gets a numeric suffix (_2, _3, …).
  • DataLore does not currently connect to remote databases (Azure SQL, Postgres, MySQL, etc.) — it only reads SQLite files on disk.
